Four Questions Every Agentic Transaction Must Answer
Identity
Who is this agent and who owns it?

Without identity, reputation, accountability, and revocation are all impossible. An agent with no verifiable owner is indistinguishable from a rogue bot.
Scoping
What is this agent permitted to do?

Without scope, an authorized agent is indistinguishable from a rogue one. Permission without boundaries is not permission — it is a blank check.
Approvals
Who authorized this, and is that proof verifiable?

Without verifiable approval, any agent can claim it was authorized. The claim is only as strong as the cryptographic proof behind it.
Enforcement
Is the authorization actually being checked at transaction time?

Without enforcement, identity, scope, and approvals are policy, not protection. The check must happen at every transaction.
"A system that answers all four is trustworthy. A system that skips any one is a liability."

BuyerBusiness buyer agents place orders but generate no structured receipt that the company's ERP can consume automatically. Reconciliation remains manual — a problem that scales negatively with agent volume.

NANDA
Open Agent Registry — Existence Without Trust
Back to concepts hub ↑
What it isNANDA is a MIT project — part of MIT's work on decentralized agent infrastructure. It was built to solve a specific gap: as AI agents proliferate, there is no shared directory where agents can find each other without going through a proprietary platform. NANDA is the open alternative. Any agent, anywhere, can register and appear in the registry without a central gatekeeper controlling access.
What it providesExistence — confirmation that an agent is registered and reachable. A buyer agent can find a seller agent's endpoint and initiate contact. What NANDA cannot tell you is whether that agent is authorized to sell, who owns it, or whether it is safe to send money to. The endpoint exists. The trust must come from somewhere else.
The gapAny agent can register. A bot, a scraper, and a legitimate Backcountry selling agent look identical in raw NANDA listings. The registry provides no quality or trust signal. A buyer agent has no basis from NANDA alone to decide whether to trust a listing enough to transact — and that gap is exactly what Firmly and LoginID are positioned to fill.
Firmly + LoginID layerFirmly adds a curated, verified layer on top of raw NANDA. A Firmly-verified listing means the agent behind it has been checked against a real merchant with accountability — and LoginID passkey binding ties that credential to a verified human owner. The model is analogous to a curated app store sitting on top of an open ecosystem — the open layer solves discovery, Firmly and LoginID solve trust. A buyer agent that trusts Firmly-verified listings can proceed with confidence that raw NANDA alone cannot provide.

Business Agent Identity
DNSid + LoginID — DNS-Anchored Agent Ownership for Businesses
Back to concepts hub ↑
Identity DigitalIdentity Digital is one of the largest domain registry operators in the world — they operate registries for over 300 top-level domains, including .info and a large family of generic extensions. They are not a startup. Their institutional weight behind DNSid means the standard has the backing of existing DNS infrastructure operators, not just a whitepaper. DNSid is an open standard that Identity Digital governs and maintains — any party can implement it, it is not a proprietary Identity Digital product.
DNSid mechanismAnchors agent ownership to a DNS domain. A business that owns a domain can publish a cryptographic record proving it owns a specific agent. No central authority needs to vouch for the relationship — the proof is as decentralized as DNS itself. Any verifier can check it independently, with no call back to Identity Digital or any other intermediary.
Revocation via DNSDNSid supports revocation by updating the DNS record. Propagation runs through DNS infrastructure — typically minutes to hours depending on TTL. No single database to corrupt, no single point of failure. This is DNSid's native revocation path. However, Firmly does not rely on DNS propagation for revocation speed. Firmly's registry is the authoritative revocation layer — when a credential is revoked, the signal is immediate at the registry. DNSid provides the identity anchor; Firmly controls when and whether an agent credential is accepted.
LimitationRequires a domain. Works for businesses and organizations that own and control a domain. Does not solve the individual human case — a consumer with no domain has no DNSid anchor. That is the gap Human Agent Identity is designed to fill, extending the same trust model to people who are not businesses.

AP2 v0.2.0
Agentic Payment Protocol — Cryptographic Scope Enforcement
Back to concepts hub ↑
What v1.0 hadThe concept of a spending mandate. A human could instruct an agent with a budget and a list of approved sellers. The problem: the mandate lived as a data structure in memory that the agent could modify. There was no cryptographic enforcement. The mandate was a convention that any sufficiently motivated agent could bypass — policy, not protection.
What v0.2.0 addsSigned Intent Mandates and Cart Mandates. The human signs the mandate upfront with a private key — approved sellers, dollar limits, categories, time window. The agent signs the cart at checkout. Both signatures are cryptographically verifiable by the seller agent independently, with no assertion required from the buyer agent about what it is allowed to do.
The chainA non-repudiable chain from human intent to payment execution: human signature covers the mandate, agent signature covers the cart, seller verifies both before completing the transaction. Every payment is traceable back to a signed human authorization. An agent cannot claim scope it was not granted because the grant is cryptographic, not conversational.
The gapThe signed mandate is visible to the seller agent. It does not travel to the issuer unless the issuer implements AP2 on their side. Network-level enforcement — where an issuer can independently verify the mandate before authorizing a card transaction — requires issuer adoption. That is the partnership conversation with a network partner like Mastercard.

x402
HTTP-Native Micropayment Rail — Monetizing Pre-Transaction Interactions
Back to concepts hub ↑
The mechanismWhen a buyer agent queries a seller agent's catalog or API, the seller returns HTTP 402 (Payment Required). The buyer agent pays a USDC micropayment — typically fractions of a cent — before the catalog query is fulfilled. The interaction is HTTP-native: no new protocol layer, no SDK required beyond the payment response handler. It fits into existing web infrastructure without modification.
What it solvesToday, seller agents provide catalog access, query responses, and pricing data at zero cost and zero friction. A competitor's buying agent can scrape a merchant's entire pricing structure for nothing. x402 introduces protective friction: unregistered or malicious agents face a cost at every query. Registered buyer agents pay a small fee that is less than the value of the data they receive.
Network notex402 settles in USDC for micro-interactions — catalog queries, data access, pre-transaction commerce. The final purchase transaction continues to run on card infrastructure. These are complementary rails, not competing ones. A network partner integrating at the final transaction layer is unaffected by x402 at the pre-transaction layer.

Seller Authorization Manifest
Merchant-Signed Authorization Boundaries — Not the Catalog
Back to concepts hub ↑
What it isA merchant-signed policy document — not a live catalog or a price feed. Think of it as the merchant's authorization letter to their agent: which categories the agent can sell from, which price ceiling it cannot exceed, which geographies and payment methods apply, valid from date X to date Y. It does not need to be updated every time inventory changes. The manifest defines the authorization envelope; the live catalog fills the content inside that envelope.
The SSL analogyAn SSL certificate does not list every file on a server. It says "this server is who it claims to be, and anything it serves is covered by this trust anchor." The Seller Authorization Manifest works the same way: "this agent is authorized by this merchant, within these boundaries." The buyer agent verifies it once at session start — like a browser checking SSL on page load — and then interacts with the live catalog knowing the counterparty is a legitimately authorized seller.
Why it mattersA buyer agent with no manifest has no way to distinguish a legitimate seller agent from a rogue one. The manifest is what turns an anonymous endpoint into a verifiable counterparty. Without it, trust is delegated entirely to the counterparty's own assertions — which is no trust at all. The manifest is signed with the merchant's DNSid key and verifiable by any party independently, with no call back to Firmly required.
The gapThe manifest establishes trust at the agent-to-agent layer. At the network level, issuers can technically consume it today — the data is available. Whether they do is a business adoption decision, not a technical gate. A network partner surfacing the manifest to issuers as part of their authorization flow is the partnership conversation, not a prerequisite for the manifest to be useful at the merchant layer.
